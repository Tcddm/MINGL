# 架构总览
:::warning
文章为AI生成

MINGL 的架构只做一件事：**把编译期的静态声明，无损地转换为运行时的动态界面**。在这条链路上，每一层都有自己的职责边界，不越界，不耦合。

## 核心哲学

MINGL 不是那种"用 C 模拟面向对象"的 GUI 框架。它走的是另一条路——**数据即界面**。

声明式宏在编译期展开为静态数据块（`mgl_widget_blueprint_t`），框架在页面加载时把这些数据块实例化为控件树。运行时不分配堆内存，不维护空闲链表，不产生碎片。所有的内存状态在编译后完全确定。

三个原则贯穿整个架构：

- **编译期确定性**：所有内存布局在 `.data`/`.bss` 段静态存在，运行时行为可预测
- **局部性**：每个控件的绘制不依赖全局状态，只依赖裁剪上下文和自身数据
- **最小接口**：子系统之间通过窄接口通信，更换实现不需要改调用方

## 从声明到像素：完整链路

理解 MINGL 最简单的方式，是追踪一段声明式代码如何变成屏幕上的像素。以 `MGL_LABEL(.text=MGL_STR("Hello"))` 为例。

### 阶段一：编译期 — 宏展开为蓝图

```c
MGL_LABEL(.text=MGL_STR("Hello"), .painter=MGL_SOLID_PAINTER_TEMP(MGL_COLOR_WHITE))
```

展开为：

```c
&((mgl_widget_blueprint_t){
    .init = mgl_label_init,
    .size = sizeof(mgl_label_t),
    .args = &(mgl_label_args_t){
        .text = "Hello",
        .painter = MGL_SOLID_PAINTER_TEMP(MGL_COLOR_WHITE)
    }
})
```

这是一个 `static const` 数据块，位于 `.rodata` 段。不涉及任何运行时逻辑。

::: tip
MGL_STR("Hello") 本身也是一个宏。它标记了文本和默认字号，字体生成工具会扫描所有 MGL_STR/MGL_FMT 调用，自动提取字符并生成位图数据。
:::

### 阶段二：页面加载 — 蓝图实例化为控件

页面切换时，`mgl_blueprint_collect` 遍历蓝图树：

```
蓝图树 (mgl_widget_blueprint_t)          控件树 (mgl_widget_t)
─────────────────────────────────        ────────────────────────
root ───▶ {init,size,args}      ──▶     root (已分配,已初始化)
  │                                        │ first_child
  ├──▶ {init,size,args}        ──▶        ├── child_0
  └──▶ {init,size,args}        ──▶        └── child_1
```

三步操作：
1. 从**页面内存池**分配 `blueprint->size` 字节
2. 调用 `blueprint->init(widget, args)` 初始化
3. 挂到父控件的 `first_child` 链表上

蓝图可以跨页面复用——同一个蓝图的 `&blueprint` 指针被多个页面引用，每次 alloc 都产生独立的控件实例。

<<< @/../../src/widget/mgl_widget_blueprint.h#mgl_widget_blueprint_t{c}

### 阶段三：布局 — 确定位置和大小

控件实例化后只有属性，没有位置。布局系统负责把每个控件的 `bounds` 计算出来。

流程：根控件→递归 `measure()` 获取自然尺寸→递归 `layout()` 分配位置。

`measure()` 的职责是回答"你希望多大"，`layout()` 的职责是"你的子控件应该放在哪里"。两者分离让容器在分配子控件位置前先知道它们的尺寸需求。

### 阶段四：渲染 — 从脏标记到像素

每帧刷新时，渲染器检查根控件的 `dirty` 标志：

| 状态 | CPU 开销 |
|------|----------|
| 脏标记未设置 | 一次 `if(w->dirty)` 判断，直接跳过整帧 |
| 脏标记已设置 | 深度遍历控件树，执行七步渲染流程 |

渲染器的核心是 `mgl_render_widget`，它用三个深度栈（clip / clr / clr_valid）做非递归深度优先遍历，对每个控件执行七个步骤。

关键优化：容器因子冒泡变脏时**不自增长脏区域**。一个子控件的局部变化不会污染整个页面。

七个步骤的完整说明见[渲染器章节](/subsystem/render)。

### 阶段五：事件 — 从输入到动作

```
HAL 触摸数据 → 命中测试(最深可见控件) → 冒泡(on_event) → 动作转换(get_action) → 回调(action_handler)
```

每一环都是虚表调用，控件按需实现。不处理的环返回 false / NULL 让框架走默认路径。

::: tip 详细事件流程
事件生命周期的完整说明见[事件系统](/subsystem/event)和[动作系统](/subsystem/action)。
:::

### 阶段六：输出 — 像素落到屏幕

所有 `draw()` 函数通过裁剪上下文（`mgl_draw_ctx_t`）画像素。`ctx->clip` 在控件树遍历时逐层缩小，保证控件不会越界绘制。

最终像素通过 HAL 的 `mgl_hal_set_pixel` 写入帧缓冲，`mgl_hal_flush_display` 刷新到屏幕。

### 依赖规则

- **HAL 是单向的底**：HAL 不依赖框架任何其他层。平台只需实现 `mgl_hal.h` 中声明的函数。
- **画笔/主题可热替换**：`draw()` 每次调用时实时解析主题画笔，切换 `g_mgl_theme` 指针后重绘即可全局换肤。
- **控件不感知平台**：控件的 `draw()` 通过裁剪上下文和 HAL 间接操作像素，不直接调用 HAL 函数。
- **日志是全局设施**：任何层都可以调用 `MGL_LOG_INFO(tag, ...)`，输出函数通过 `mgl_log_set_output` 注入。

## 关键设计决策

### 为什么用蓝图而不是直接 new

传统 GUI 框架用 `create()` 函数动态分配控件。MINGL 用蓝图的原因：

- **编译期确定内存用量**：蓝图树生成后，`mgl_blueprint_collect` 遍历时就能算出总内存。页面池大小可以在 Kconfig 里调到刚好够用。
- **蓝图复用**：同一份蓝图可以被多个页面引用，省代码量。
- **无运行时类型信息**：`init` 函数指针替代了虚表注册。不需要 RTTI，不需要 vtable 在堆上。

代价：不能运行时动态创建控件。这是有意为之——MINGL 面向 MCU 场景，界面结构在编译时已知。

### 为什么脏标记沿父链冒泡

```c
static inline void mgl_widget_set_dirty(mgl_widget_t *w) {
    while(w){
        w->dirty=1;
        w=w->parent;
    }
}
```

不冒泡到根，渲染器就不会触发。冒泡的同时把脏容器标记为"因子冒泡变脏"，渲染器据此决定 Step 3 是否把容器自身加入脏区域。

::: warning
调用 `mgl_widget_set_dirty` 后，控件会在**下一帧**才被重绘，不是立即绘制。如果有多个控件在同一帧内被标记脏，它们会在同一帧内一起处理。
:::

### 为什么裁剪用传递式而非全局合并

传统方案：收集所有脏矩形→合并→在该区域重绘所有相关控件。缺点：合并引入过度绘制。

MINGL 方案：裁剪矩形随控件树遍历逐层求交传递。每个控件收到的 `ctx->clip` 已经是"你可以合法绘制的像素区域"。不需要全局脏矩形列表，不需要合并。

代价：深度遍历的开销。控件树很深时每帧遍历成本不为零。但 MINGL 面向的场景控件树深度通常在 3-5 层，这个成本可以忽略。

### 为什么事件用冒泡而非广播

冒泡（从目标控件沿 parent 链向上）比广播（向所有控件发送）更精确。一个触摸事件只有一个目标控件，只有它和它的祖先需要知道这个事件。

MINGL 的冒泡还附加了动作转换：`on_event` 返回后框架自动调用 `get_action` 和 `action_handler`。这意味着控件只需要关心事件处理（如触摸按下时变色），响应逻辑（如按钮点击时执行业务代码）放在 action_handler 里，完全分离。

### 为什么页面用栈式而非堆式

页面切换是栈式的（push/pop），不是随机的（任意跳转）。所以页面内存池可以用栈式分配：push 时 alloc，pop 时整块释放，不需要空闲链表管理碎片。

`mgl_page_pool_reset` 一键清空整块内存——比逐个 free 快，且保证下次 alloc 时内存是干净的。

## 数据流总览

```
用户声明 DSL 宏
      │
      ▼ (编译期)
  静态蓝图树 (mgl_widget_blueprint_t, .rodata)
      │
      ▼ (页面加载: mgl_blueprint_collect)
  控件树 (mgl_widget_t 链表, 页面池)
      │
      ├──▶ 布局 (measure → layout) ──▶ bounds 确定
      │
      ├──▶ 渲染 (dirty → clip_stack 遍历 → draw) ──▶ 像素 → HAL
      │
      └──▶ 事件 (命中测试 → 冒泡 → 动作) ──▶ action_handler 回调
```

这条链路是单向的：声明→蓝图→控件树→像素。事件路径是反向的：输入→控件树→回调。两条路径共享同一棵控件树，但互不干扰。
